# Cellproject — Concept

## One-line pitch

Cellproject is a **visual planning framework based on decomposition**: the user declares a concrete goal, the AI proposes complete paths shaped as a tree, and the user validates node by node until it ships.

## The problem it solves

Planning something new (building, assembling, learning) stalls at two points:

1. **Cold-start paralysis** — I don't know where to begin, what already exists, or what comes before what.
2. **Context loss during execution** — the plan becomes an opaque list, each item becomes a blind task, the *why* behind every step disappears.

Traditional tools (to-do lists, kanbans, free-form mind maps) only solve half the problem: they either structure poorly or structure without teaching. Cellproject assumes that planning and learning are the same activity — the tree that guides execution is also the didactic material.

## Principles

- **Goal first, always.** Nothing starts without a `f(X) = ?` written down. The goal is the root.
- **Typed decomposition.** Every node has one of a fixed set of roles: `recurso`, `passo`, `decisao`, `categoria`, `concept`. The kind defines how the AI decomposes it.
- **Reuse-first, but the user chooses.** The AI is biased to reach the goal by reusing, adapting and recombining what already exists rather than forging each part from scratch — but the *construction strategy* (`reaproveitar` / `hibrido` / `do_zero`) is an explicit choice the user makes when picking a plan, and it conditions every later decomposition. From-scratch is always available, never forced.
- **Every node is validatable on its own.** A node carries `oQue` (what), `porQue` (why), `comoConfirmar` (how to confirm). If the user can confirm, they advance. If not, they decompose further.
- **AI is co-author, not owner.** It proposes plans and sub-trees in streaming; the user accepts or rejects per node or in batch. Suggestions sit in staging until confirmed.
- **The loop can never close inside AI alone.** A criterion generated by the same AI that wrote the plan is theater. Every node needs at least one anchor outside the AI: a user-written criterion, an adversarial critique, a verifiable real-world reference, or a reported failure. This is now **enforced at conclusion**: a node only reaches `done` when it carries real signal (see *Conclusion gate*). Without it, confirming still advances the flow but the node stays `validated`, not `done`.
- **Same structure, two views.** `graph` to see the whole and connect pieces; `tutor` to dive into a node and execute it.
- **Direct tone, no fluff.** Reflected in the UI and in the system prompts. "2 bamboo sticks 40cm" beats "some materials".

## Mental model

A **project** is a tree with metadata:

```
Project (goal)
└── root
    ├── category: Resources
    │   ├── resource A
    │   └── resource B
    ├── category: Execution
    │   ├── step 1
    │   ├── step 2
    │   └── step 3
    └── category: Decisions            (optional, only when there is a real trade-off)
        └── decision X (option A | B | C)
```

### Node kinds (`NodeKind`)

| Kind        | Role                                                                  |
| ----------- | --------------------------------------------------------------------- |
| `root`      | The goal itself.                                                      |
| `categoria` | Container. Always one of three: Resources, Execution, Decisions.      |
| `recurso`   | Something that must be available before execution.                    |
| `passo`     | Concrete action, chronologically ordered (`order`).                   |
| `decisao`   | Choice between mutually exclusive paths (`decisionOptions`).          |
| `concept`   | Auxiliary node — use only when the others don't fit.                  |

### Node state (`NodeState`)

State is **earned, never painted** — there is no manual state picker in the UI.

- `concept` — created, untouched.
- `validated` — confirmed by the user, but without real signal (see *Conclusion gate*).
- `done` — confirmed **with** real signal (or the explicit, audited `force` opt-out).
- `problem` — set by `reportFailure` (mechanism c), always carrying `failureContext`.
- `executing` / `discarded` — legacy values still rendered for old data; the UI no longer sets them.

The boolean `confirmado` marks whether the user confirmed the node. **Reaching `done` is gated** (see *Conclusion gate*): a node earns `done` only against real signal — a locked user criterion or a verified anchor — and only when not blocked by an earlier sibling. Without signal, confirming still sets `confirmado: true` and advances the flow, but the node stays `validated` and is flagged `confirmedWithoutSignal` (no green / 100%). The selector `canConcludeNode(project, nodeId)` is the single source of this rule.

### Progress = the decomposition frontier

Progress counts **effective leaves**: every actionable node (`recurso`, `passo`, `decisao`, `concept` — in any archetype) that has no children, plus every `takenAsKnown` node (the recursion floor — whatever lies below a floor stops counting). A decomposed node is a container: it resolves through its children, which is also what unblocks the next step after you broke the previous one into sub-steps instead of confirming it directly. The selectors `effectiveLeaves`, `isNodeResolved` and `nextPendingForTutor` all walk the tree the same way, in sibling `order` (the decomposition order), depth-first.

### Edges (`EdgeKind`)

- `direct` — A directly precedes B (step N → step N+1).
- `middleware` — A influences B via an intermediary node.
- `independent` — informational link, no dependency.
- `optional` — alternative path.

### Key fields per node

- **Conceptual**: `fx` (function/purpose), `problem`, `pros`, `cons`, `confidence` (0–100 + `confidenceSource`, `confidenceReason`) — confidence is the AI's signal, displayed with its source and reason, never edited by hand: the user's truth enters through the ground-truth mechanisms.
- **Didactic**: `oQue`, `porQue`, `comoConfirmar`, `explicacao` (generated on demand by the tutor).
- **Ground truth** (see dedicated section): `comoConfirmarUsuario` + `comoConfirmarUsuarioAt`, `critica`, `groundTruthRefs`, `failureContext` + `failureReportedAt`.
- **Execution**: `confirmado`, `order`.
- **Decision**: `decisionOptions`, `decisionPickedId`.
- **Audit**: `history` (timeline of changes), `aiSuggested`.

> Implementation note: several field names are still in Portuguese (`oQue`, `porQue`, `comoConfirmar`, `confirmado`, etc.) — they predate the language switch and remain as domain identifiers. The UI and the system prompts are bilingual.

## Flows

### 1. Project creation — `ObjectiveScreen`

1. The user types a name and writes the goal.
2. `generatePlans(objective)` requests 1–3 alternative plans — each a different **construction strategy** (`reaproveitar` / `hibrido` / `do_zero`), reuse-led by default — via streaming (`streamObject`).
3. The screen shows the plan assembling live (phase `connecting` → `streaming` → `finalizing` → `done`) and labels each card with its strategy.
4. The user consciously picks a plan (no auto-selection). `createProjectFromPlan` materializes the tree and persists the chosen `constructionStrategy`, which conditions later `decomposeNode` / `replanFromFailure`.

### 2. Exploration and expansion — `GraphCanvas` + `DetailPanel`

1. Graph view (React Flow) with lenses: `structure`, `flow`, `risk`, `state`, `connections` (keys 1–5).
2. Click a node → `DetailPanel` shows metadata, notes, history.
3. Decompose a node → `decomposeNode(ctx)` generates children + edges based on the kind (see `decomposeGuidance` in `src/ai/service.ts`).
4. Suggestions sit in **staging** (`pendingSuggestions`): `accept` / `reject` per node or in batch. They enter the graph only after acceptance.

### 3. Guided execution — `TutorMode`

The tutor is the concept's core loop made into a screen.

1. Key `t` enters tutor mode; `g` returns to the graph.
2. The sidebar **is the decomposition tree**: categories and decomposed nodes appear as groups with their own progress; leaves (`recurso` / `passo` / `decisao` / `concept`) are rows in decomposition order. Clicking a row dives into that part; the checkbox one-click-confirms only nodes that already carry real signal — without signal it routes to the card, where confirming is a deliberate two-step. The unfaithful path is never the easiest click.
3. The card shows ONE leaf — the one you picked, or the next pending in **depth-first** order: after decomposing a node, its parts come before the next sibling, so you go down to the atom before moving on.
4. The card presents `oQue` / `porQue`, the ground-truth block, and the fork as actions:
   - **Confirm** ("I have it / I did it / I understood it") — gated by `canConcludeNode`.
   - **"Can't confirm yet → break it into parts"** — `decomposeNode`, staged inline for accept/reject.
   - **"I already know this"** — `takenAsKnown`, the recursion floor; closes the whole subtree.
5. `explainNode(ctx)` generates, on demand, a long markdown explanation (senior-engineer depth, structured with **UPPERCASE BOLD** headings).

### 4. Decisions

When a node is `decisao`, the user picks one of `decisionOptions` — in the tutor card or in the detail panel; the pick **is** the decision's confirm and goes through the same conclusion gate. It becomes history (`kind: 'decision'`) and sets `decisionPickedId`, which can condition the next steps.

## Closing the loop — ground truth

> The biggest conceptual risk in Cellproject is the **closed AI→AI loop**: the same AI generates the plan and the confirmation criterion, the user confirms, and no signal from reality enters. In exchange, four orthogonal mechanisms ensure that every node can be anchored outside the AI. None is mandatory in isolation, but a node without any of them is theater.

### (a) User criterion, locked before seeing the AI's

Field `comoConfirmarUsuario` + timestamp `comoConfirmarUsuarioAt`. The user writes how they would know the node is done **before** the UI reveals the AI's `comoConfirmar`. After writing, the field is locked (`setUserCriterion` refuses overwrite when `comoConfirmarUsuarioAt` is set). That prevents the person from retroactively copying the AI's criterion and believing they agree.

### (b) Adversarial critique — second skeptical pass

Function `critiqueNode(ctx)` in `src/ai/service.ts`, with `criticSystem()` distinct from the planner's `baseSystem()`. Returns `{ fraquezas, premissasOcultas, criterioAlternativo }`. The real value is in `criterioAlternativo`: an independent criterion written as if the evaluator were a skeptic — if it converges with the original `comoConfirmar`, the plan is robust; if it diverges, the user has a signal that something deserves attention.

### (c) Replan from real failure

When execution doesn't work, the user calls `reportFailure(id, context)` — the node becomes `problem`, `confirmado` goes back to `false`, and `failureContext` keeps what actually broke. The UI can then call `replanFromFailure(ctx)` in `src/ai/service.ts`, which returns a new decomposition **knowing** about the failure and with explicit instructions not to repeat the broken premise. The output flows through the same staging mechanism as `decomposeNode` — the user accepts / rejects.

### (d) Verifiable real-world anchors

Field `groundTruthRefs: GroundTruthRef[]`. Each anchor has `kind` (`link`, `spec`, `medida`), `label`, `value`, `verificado`, and `addedByAI`. The AI already suggests anchors via `groundTruthHints` while generating plans or decomposing nodes — but they come with `verificado: false` and `addedByAI: true`. They become truth only after the user checks in the real world and calls `toggleGroundTruthVerified`. The user can also add their own anchors (`addedByAI: false`) via `addGroundTruthRef`. Examples: `spec: "Phyllostachys aurea, 40cm ± 2cm, Ø 5-8mm"`, `medida: "weight < 15g"`, `link: URL of a known tutorial`.

### How the four interact

| Mechanism              | Who starts       | When it runs              | Breaks the loop because…                                |
| ---------------------- | ---------------- | ------------------------- | ------------------------------------------------------- |
| User criterion         | User             | Before seeing the AI's    | The person isn't just validating — they're asserting.   |
| Adversarial critique   | User → AI        | On demand, per node       | A distinct persona generates an independent criterion.  |
| Replan from failure    | User → AI        | When reality refuses      | Real failure context is the input the AI doesn't have.  |
| Verifiable anchors     | AI + User        | In the plan, continuously | Each `verificado: true` is one bit of reality entering. |

### Conclusion gate (`canConcludeNode`)

The four mechanisms above are no longer optional decoration at the moment that matters. A node reaches `done` (the green / 100% state) **only** when it carries real signal — a locked user criterion (a) or a verified anchor (d) — and isn't blocked by an earlier sibling. The store selector `canConcludeNode(project, nodeId)` encodes this rule, and both `confirmNode` and `pickDecisionOption` route through it. A confirm without signal still advances (so the flow is never a nagging wall) but stays `validated` + `confirmedWithoutSignal`; `confirmNode(id, { force: true })` is the explicit, audited opt-out for "I'm sure, no anchor". Two UI supports reinforce this: the AI's `comoConfirmar` is revealed only **after** the user locks theirs (no peek-before-lock), and the divergence hint **warns** when the user's criterion merely echoes the AI's instead of rewarding the copy.

## Architecture

- **Pure frontend** (no proprietary backend). Vite + React 18 + TypeScript.
- **Graph**: `@xyflow/react`.
- **State**: Zustand with `persist` + IndexedDB (`idb-keyval`) — projects survive reloads without a server.
- **AI**: Vercel AI SDK (`ai`) + `@ai-sdk/openai`, `@ai-sdk/anthropic`, `@openrouter/ai-sdk-provider`. Provider and model selected by the user via BYOK at runtime.
- **Schemas**: Zod (`PlansResponseSchema`, `DecomposeResponseSchema`, …) ensure the model's response fits the domain before hydration.
- **IDs**: `nanoid` everywhere a local identity is needed.
- **Style**: Tailwind with a custom palette (`bg-primary`, `ai-accent`, etc.).
- **i18n**: handcrafted bilingual messages bundle (en / pt-BR), selectable at runtime. System prompts include "respond in {language}" so the model output follows the UI locale.

### Model contract

Five public functions in `src/ai/service.ts`. All use `baseSystem()` except `critiqueNode`, which uses `criticSystem()` (a distinct skeptic persona — that's what makes the second pass actually adversarial).

| Function              | System         | Return                                  | Use                                              |
| --------------------- | -------------- | --------------------------------------- | ------------------------------------------------ |
| `generatePlans`       | BASE           | `AIPlan[]` (streaming via callback)     | Initial screen — 1 to 3 alternative plans        |
| `decomposeNode`       | BASE           | `{ nodes, edges }` validated by schema  | Expand an existing node                          |
| `explainNode`         | BASE + tutor   | `string` (markdown)                     | Tutor mode content                               |
| `critiqueNode`        | **CRITIC**     | `AdversarialCritique`                   | Ground truth (b) — break the node as a skeptic   |
| `replanFromFailure`   | BASE           | `{ nodes, edges }` validated by schema  | Ground truth (c) — re-decompose with real failure|

The model **never** receives real project ids — it uses short `tempId`s that are mapped to `nanoid`s through `idMap` during hydration. Total isolation between the model's language and the internal domain.

The planner (`generatePlans` / `decomposeNode` / `replanFromFailure`) is instructed to populate `groundTruthHints` whenever a natural anchor exists (a measurement with unit, a concrete spec, a known link). Hints enter the node as `groundTruthRefs` with `verificado: false` — it's the user who turns each hint into truth by checking in the real world.

## What Cellproject is **not**

- It's not a generic task manager — the structure is a tree with fixed kinds, not a free-form list.
- It's not an open brainstorming tool — everything anchors on the root goal.
- It's not a chat — the AI replies in validated objects, not free text (except in tutor mode).
- It has no multi-user, real-time collaboration, or backend. A project lives in the browser's IndexedDB.

## Quick glossary

- **f(X)** — internal jargon for "what's the function of this node", inherited from the `fx` field. At the root, it's the goal; inside, it's the node's contribution.
- **Lens** — graph view mode. Same tree, different emphases.
- **Staging** — area where AI suggestions wait for acceptance before becoming part of the project.
- **Breadcrumb** — path from root to current node, passed to the AI as context in `decomposeNode` and `explainNode`.
- **BYOK** — Bring Your Own Key. The user supplies the API key; it lives only in the browser's IndexedDB.
