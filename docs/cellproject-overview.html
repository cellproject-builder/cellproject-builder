◆ Cellproject · living overview

Planning and learning, on the same tree.

Cellproject is a visual planning framework based on decomposition. The user declares a concrete goal, the AI proposes complete paths shaped as a tree, and the user validates node by node until it ships — using the plan itself as didactic material.

Beta Current version: 0.1.0 Stack: Vite + React 18 + TypeScript Persistence: Zustand + IndexedDB UI: bilingual (en / pt-BR)

01 · Pitch and problem

What Cellproject solves

Planning something new (building, assembling, learning) stalls at two classic points:

  1. Cold-start paralysis — I don't know where to start, what exists, or what comes before what.
  2. Context loss during execution — the plan becomes an opaque list, each item becomes a blind task, the rationale for each step disappears.

To-do lists, kanbans, and free-form mind maps only solve half the problem: they either structure poorly or structure without teaching. Cellproject assumes that planning and learning are the same activity — the tree that guides execution is also the didactic material.

Nothing starts without a written f(X) = ?. The goal is the root; everything below only exists to contribute to it.

02 · Principles

What keeps the framework coherent

◆ Goal first

Always a root

No floating node. Every decomposition pulls a f(X) declared by the user at the top.

◆ Typed decomposition

5 fixed roles

Every node is recurso, passo, decisao, categoria, or concept. The kind drives how the AI decomposes it.

◆ Validatable in isolation

oQue / porQue / how to confirm

If the user can confirm via the criterion, they advance. If not, they decompose further.

◆ AI is co-author

Per-node acceptance

The AI proposes plans and sub-trees via streaming. Suggestions sit in staging until a human confirms.

◆ Loop never closes inside AI

Every validation needs an external anchor

A criterion generated by the same AI that wrote the plan is theater. Four mechanisms exist to force ground truth — see section 07.

◆ Direct tone, bilingual

"2 sticks 40cm", not "some materials"

Specific beats generic. Applies to UI and to the model's system prompts. UI is available in English and Brazilian Portuguese.

03 · Mental model

A project is a tree with metadata

The canonical shape is this:

Project (goal) └── root (f(X) = ?) ├── category: Resources │ ├── resource A │ └── resource B ├── category: Execution │ ├── step 1 │ ├── step 2 │ └── step 3 └── category: Decisions (optional, only when there is a real trade-off) └── decision X (option A | B | C)

Each node carries concept, didactic, ground truth, execution, and audit data. All fields live in ConceptNodeData (src/types.ts) and are populated either by the AI, by the user, or by both with clear locking semantics.

Naming note: some field names are still in Portuguese (oQue, porQue, comoConfirmar, confirmado, recurso, passo, decisao, categoria). They predate the bilingual UI and remain as domain identifiers — renaming them would force a data migration.

04 · Node taxonomy

NodeKind — 6 roles

kindRoleAI decomposition
rootThe goal itself.Not decomposed — it is the root.
categoriaContainer (Resources / Execution / Decisions).3–8 children of the natural kind for the category.
recursoSomething available before execution.2–4 sub-resources or acquisition stages.
passoConcrete action, ordered (order).2–5 sub-steps chained with direct edges.
decisaoChoice between mutually exclusive paths.2–4 nodes detailing the post-decision path.
conceptAuxiliary node — use only when others don't fit.2–5 children of the most adequate kind.

05 · State and edges

NodeState and EdgeKind

Lifecycle: concept → validated → executing → done. At any moment it can become problem or discarded. The boolean confirmado is orthogonal and marks whether the user has cleared the comoConfirmar check.

EdgeKindMeaning
directA directly precedes B (step N → step N+1).
middlewareA influences B through an intermediary node.
independentInformational link, no dependency.
optionalAlternative path.

In the graph (GraphCanvas) each EdgeKind has its own color and dasharray: direct animated green, middleware dashed amber, independent dotted indigo, optional light gray.

06 · Main flows

Three journeys, one project

1. Creation — ObjectiveScreen

  1. User types a name and the f(X) (goal).
  2. generatePlans(objective) requests 1–3 plans from simplest to most ambitious, via streaming streamObject.
  3. The screen shows the plan assembling live (connecting → streaming → finalizing → done).
  4. User picks a plan → createProjectFromPlan materializes the tree in Zustand.

2. Exploration and expansion — GraphCanvas + DetailPanel

  1. Graph in React Flow with 5 lenses (keys 1–5): structure, flow, risk, state, connections.
  2. Click a node → DetailPanel opens with metadata, notes, history.
  3. "Detail with AI" button → decomposeNode(ctx) generates children + edges based on the kind.
  4. Suggestions fall into staging (pendingSuggestions). Each becomes a dashed node in the graph with accept / reject buttons.

3. Guided execution — TutorMode

  1. Key t enters tutor mode; g returns to graph.
  2. nextPendingForTutor() picks the next node: unconfirmed resources first, then steps respecting order within each Execution category.
  3. explainNode(ctx) generates, on demand, a long markdown explanation (structured with UPPERCASE BOLD headings).
  4. User confirms → confirmNode(id) advances to the next.

4. Decisions

When a node is decisao, the user picks one of decisionOptions. The pick becomes history (kind: 'decision') and sets decisionPickedId, which can condition the next steps.

07 · Ground truth

The 4 mechanisms that break the AI→AI loop

The biggest conceptual risk in Cellproject is the same AI generating the plan and the confirmation criterion, the user confirming, and no signal from reality entering. Four orthogonal mechanisms ensure that every node can be anchored outside the AI.
MechanismWho startsWhenWhy it breaks the loop
(a) User criterion User Before seeing the AI's The person doesn't just validate — they assert. Field comoConfirmarUsuario locks after setUserCriterion.
(b) Adversarial critique User → AI On demand, per node A criticSystem() persona, distinct from baseSystem(), generates an independent criterioAlternativo.
(c) Replan from failure User → AI When reality refuses reportFailure(id, context) feeds replanFromFailure with what the world proved wrong.
(d) Verifiable anchors AI + User In the plan, continuously groundTruthRefs (link / spec / medida). Each verificado: true is one bit of reality entering.

Important detail of (a)

setUserCriterion in store.ts refuses overwrite once comoConfirmarUsuarioAt is set. And UserCriterionField hides the AI's comoConfirmar until the user locks theirs — preventing retroactive copy.

Important detail of (d)

The AI already suggests anchors via groundTruthHints. But they enter with verificado: false and addedByAI: true. They become truth only after the user checks in the world and calls toggleGroundTruthVerified.

08 · Current architecture

Stack today

Frontend

Vite + React 18 + TS

Pure SPA, no SSR. Tailwind 3 with a custom palette (bg-primary, ai-accent, conf-high…).

Graph

@xyflow/react 12

Custom layout in layoutFromPlan; 5 lenses recompute positions in lensPositions.

State

Zustand + IndexedDB

persist + idb-keyval. Migration with versioned store keys.

AI

Multi-provider BYOK

Vercel AI SDK + @ai-sdk/openai · @ai-sdk/anthropic · @openrouter/ai-sdk-provider. User chooses provider and pastes a key — it lives in IndexedDB.

Validation

Zod schemas

PlansResponseSchema, DecomposeResponseSchema, CritiqueResponseSchema. The model never receives real ids — uses tempIds mapped via idMap.

Local PDF

pdfjs-dist in browser

extractPdfText runs 100% on the client. The binary never leaves the device; only the extracted text is sent for summarization.

i18n

Handcrafted en / pt-BR

Single messages.ts with both bundles, switched via locale store. System prompts include "respond in {language}" so model output follows the UI locale.

Deploy

Railway-ready

railway.json + nixpacks.toml serve dist/ via serve -s on Node 22. No mandatory env vars — BYOK lives client-side.

09 · Model contract

Five functions in src/ai/service.ts

FunctionSystemReturnUse
generatePlansBASEAIPlan[] via streamingInitial screen — 1 to 3 plans
decomposeNodeBASE{ nodes, edges } validatedExpand a node
explainNodeBASE + tutormarkdown stringTutor content
critiqueNodeCRITICAdversarialCritiqueGround truth (b)
replanFromFailureBASE{ nodes, edges } validatedGround truth (c)

All in the user's active language (driven by "respond in {language}" in the system prompt). criticSystem() is deliberately skeptical — "no 'great point, but'". The model never sees real project ids: it uses short tempIds mapped to nanoids during hydration.

10 · Knowledge base

User PDFs become cheap context for the planner

The KB is the framework's newest leverage point. It works in 3 stages:

  1. Upload — KnowledgeRepo receives PDFs. extractPdfText runs via pdf.js in the browser; fingerprintText (djb2) deduplicates.
  2. Summarize — summarizeDocument calls the cheaper kbModel and returns KBSummary: titulo, dominio, tags, resumo (3–6 bullets), fatos (verifiable measurements / specs / rules / links), nivel.
  3. Pick — before any large prompt, getContextFor({ label, extra }) calls pickRelevantDocs (strict AI judge, up to 2 docs). The result is cached by context hash in pickCache.

Each injected doc weighs ~400–800 tokens — summary bullets + up to 5 facts. formatKBContext in ai/service.ts assembles the KNOWLEDGE BASE block at the start of the user prompt and instructs the model to cite [docId].

Where KB is wired today: generatePlans, decomposeNode, replanFromFailure. Not wired yet: critiqueNode (could strengthen the counter-anchor) and explainNode (could ground the tutor's prose).

11 · Localization (i18n)

Single bundle, two languages, runtime switch

The i18n is intentionally simple: no library, no fetched JSON, no key-by-key lookup at runtime. Both bundles live in src/i18n/messages.ts as plain TypeScript objects. The shape is enforced by TypeScript (type Messages = typeof en), so adding a string to en without a counterpart in ptBR is a compile error.

  • Default locale: detected from navigator.language on first run (anything starting with pt picks pt-BR, otherwise en).
  • Switching: LanguageSelector in ApiKeyGate persists the choice in IndexedDB (cellproject-locale).
  • AI integration: currentLocaleAIName() returns "English" or "Brazilian Portuguese"; baseSystem() and criticSystem() inject that string into "Always respond in {lang}". The user sees plans, decompositions, and tutor explanations in the chosen language.
  • Domain identifiers stay in Portuguese (recurso, passo, oQue, …). The UI labels them via tr.conceptNode[kind] so users never see the raw identifier.

13 · What's not there yet

Concrete gaps — not promises

  • partialMultiple local projects. Today the store keeps one project (project | null) — "Reset" discards. No list, no switch.
  • partialKB wired everywhere. critiqueNode and explainNode still don't receive kbContext. Cited [docId]s aren't clickable yet.
  • plannedPer-node reasoning UI. extractReasoning already captures provider reasoning in generatePlans, but decomposeNode doesn't surface it.
  • plannedBatch adversarial critique. Today it's per node. A "critique all leaves" sweep would surface weak spots across a category.
  • plannedImport / device sync. JSON export exists, but no import flow or reconciliation with the local store.
  • plannedDomain templates. Today every plan starts from scratch. A "start from X" template would beat KB hunger for popular domains.
  • plannedMobile / PWA tutor standalone.
  • plannedOnboarding teaching "write your criterion before seeing the AI's" on first session.

14 · Open risks

Things that bite if left undecided

◆ Conceptual

AI→AI loop creeping back

The 4 mechanisms exist, but today nothing forces the user to use any of them. A node marked done with no locked criterion or verified anchor is still theater. Open call: require 1 of the 4 before allowing confirmado: true?

◆ Technical

Single-project persistence

"Reset" discards the project. An accidental click loses everything. Short mitigation: archive last discarded project with a TTL in IndexedDB.

◆ Cost

Re-decompose burns prompts

decomposeNode always generates fresh context. Without a cache of prior decompositions, the user pays every time they revisit an already-expanded node.

◆ UX

Staging with no timeout

If the user neither accepts nor rejects a pendingSuggestions batch, the "Detail" button stays locked. No TTL, no auto-dismiss.

◆ Product

High first-time curve

"Write your criterion before seeing the AI's" is powerful but requires the user to grasp why. Without onboarding it's easy to skip the friction and miss the whole point.

◆ Security

BYOK on XSS-prone domain

The API key lives in IndexedDB. Any XSS reads it. Hosting on a clean domain and keeping deps fresh is mandatory — and worth documenting clearly for users.

16 · Future possibilities

Vectors that go beyond the original thesis

◆ Educational

Cellproject as a living book

A finished project becomes a reproducible tutorial. With verified anchors, it beats a blog post — because it teaches and lets the next learner break the node where they fail, with replan.

◆ Research

Logs as dataset

Each node's history is a fine trace of what humans and AI proposed, critiqued, and validated. At scale, it's rich material to study where models fail at planning.

◆ Product

Verticals (maker, kitchen, code)

The recurso/passo/decisao taxonomy covers physical domains well. A "Cellproject for cooking" would be the same core with pre-loaded anchors (measurements in grams, times).

◆ Integration

KB beyond PDFs

Today the KB ingests only PDFs. Extending to video transcripts, code repos, or bookmarks with their own extractors widens the verifiable base without changing the contract.

◆ Model orchestration

Multi-model by role

One model for planning, a cheaper one for the KB. Next: a reasoning model (o3, Opus) reserved for critiqueNode, where extra cost pays in diagnosis quality.

◆ UX

"Plans itself" mode

Auto-loop critiquing, replanning, and converging — useful to see where the model is consistent and where it contradicts itself. Turns the tool into a plan stress-tester.

17 · Open decisions

Not to be answered now — just not forgotten

  1. Multi-project before or after a real backend? Local multi-project is easy; a backend solves it "for free" if we add one. Doing both sequentially is wasteful.
  2. Confirm requires ground truth? Forcing at least 1 of the 4 mechanisms before marking done defends the thesis, but can irritate the user on obvious nodes ("picked up the measuring tape").
  3. Reasoning as a first-class citizen or debug-only? Today it appears in ObjectiveScreen as "model thinking". Could become permanent per-node history.
  4. KB privacy when shared. Personal PDFs with PII flow to the model in the summary. In a multi-user world, that becomes a problem fast.
  5. Naming and branding. The current palette is strong and functional, but the product has no public name beyond the repo. Branding is open.
